# Enterprise Maturity Gap Analysis

本文档列出当前前端项目与"标准企业级前端基座"的差距，按优先级排列。用于指导后续架构演进方向。

## 成熟度总体评价

- 当前状态：企业级可用基座 v1
- 完成度：75% ~ 80%
- 定位：已脱离 demo 层，适合支撑 3~5 人团队长期开发；距离"10+ 人协作规范化基座"还有差距

## 差距清单

### 第一优先级（关键路径，建议立即补齐）

#### 1. 模块注册与路由平台化
**优先级**: ⭐⭐⭐⭐⭐  
**复杂度**: 中  
**工作量**: 2~3 天  

**现状**：
- 路由在 `src/App.tsx` 硬编码
- 菜单在 `src/components/Layout/AppLayout.tsx` 硬编码
- 权限码分散在各页面和菜单里
- 新增模块需要改 App.tsx、AppLayout.tsx、权限中心三个地方

**应该做什么**：
- 建立 `src/config/modules.ts`，定义模块元信息（路由、菜单、权限、权限码）
- 从模块清单自动生成路由与菜单
- 新增模块只需在 modules.ts 里补一条配置

**收益**：
- 新增业务模块成本从 20min 降到 5min
- 避免路由与菜单不同步
- 为多租户模块动态加载预留接口

**参考实现**：
```typescript
// src/config/modules.ts
export const moduleRegistry: ModuleConfig[] = [
  {
    id: 'chat',
    path: '/chat',
    label: 'uiText.layout.menuChat',
    icon: MessageOutlined,
    permission: 'menu.chat',
    component: lazy(() => import('@/pages/Chat/ChatPage')),
  },
  {
    id: 'documents',
    path: '/documents',
    label: 'uiText.layout.menuDocument',
    icon: FileTextOutlined,
    permission: 'menu.document',
    component: lazy(() => import('@/pages/Document/DocumentPage')),
  },
  // 管理后台作为一组子模块
  {
    id: 'admin',
    label: 'uiText.layout.menuAdmin',
    icon: DashboardOutlined,
    permission: 'menu.admin',
    children: [
      {
        id: 'admin.dashboard',
        path: '/admin/dashboard',
        label: 'uiText.layout.menuDashboard',
        component: lazy(() => import('@/pages/Admin/Dashboard')),
      },
      // ...
    ],
  },
]
```

#### 2. 全量页面文案国际化收口
**优先级**: ⭐⭐⭐⭐⭐  
**复杂度**: 低  
**工作量**: 1~2 天  

**现状**：
- `src/config/uiText.ts` 已支持多语言架构
- 核心页面文案已迁移
- 管理后台仍有硬编码中文（Dashboard 示例数据、部分标签）

**应该做什么**：
- 遍历代码，把所有硬编码中文字符串迁到 uiText
- 特别关注示例数据、文案、标签、tooltip
- 补齐英文翻译

**收益**：
- 真正实现一键切语言
- 降低后续本地化维护成本

#### 3. 非 CRUD 页面模板化
**优先级**: ⭐⭐⭐⭐  
**复杂度**: 中  
**工作量**: 3~5 天  

**现状**：
- Chat、Document、Agent 等特殊页面各自一套风格
- 没有可复用的"富交互页面"模板
- 新增类似页面时风格容易不一致

**应该做什么**：
- 抽取 Chat、Agent、Document 页面的通用骨架（侧边栏 + 编辑区 + 操作栏）
- 形成"搜索 + 列表 + 详情"、"编辑器 + 日志"等常见模式
- 补充模板文档

**收益**：
- 新增特殊页面开发时间缩短 30%
- 团队新人能更快上手

#### 4. Query Key 与失效规则文档
**优先级**: ⭐⭐⭐⭐  
**复杂度**: 低  
**工作量**: 1 天  

**现状**：
- Query Key 在 `src/config/queryKeys.ts` 已集中管理
- 但什么时候触发失效、什么时候用 silentError，规则不明确
- 新人容易缓存错误或失效错误

**应该做什么**：
- 补充 queryKeys.ts 的文档注释
- 写一份"缓存与失效规则"指南
- 补充常见场景的 hook 写法

**收益**：
- 降低新手搞破坏的风险
- 加速多人协作

**收益**：
- 提升用户体验稳定性
- 减少网络问题导致的数据丢失

#### 7. 权限模型强化与权限级别

## 优先级推荐路线

### 如果目标是"稳定支撑 3~5 人团队" （推荐）
1. 模块注册平台化（第一优先级 #1）
2. 全量页面文案收口（第一优先级 #2）
3. Query Key 文档（第一优先级 #4）
4. 自动化测试基础（第二优先级 #5）

**预计工作量**: 6~8 天  
**ROI**: 最高

### 如果目标是"为后续扩展到 10+ 人团队做准备"
1. 第一优先级所有项（#1~#4）
2. 非 CRUD 模板化（第一优先级 #3）
3. 权限模型强化（第二优先级 #7）
4. 错误恢复策略（第二优先级 #6）
5. 自动化测试（第二优先级 #5）

**预计工作量**: 14~18 天  
**ROI**: 高

### 如果目标是"打造对标企业级开源项目的产品"
- 完成上述两条路线
- 加上第三优先级的部分关键项（#9、#10、#12）
- 补充完整的架构文档、最佳实践文档、故障排查指南

**预计工作量**: 25~35 天  
**ROI**: 中（主要是品牌与生态收益）

## 各项实施指南

### 1. 模块注册平台化

**建议步骤**：
1. 在 `src/config/modules.ts` 定义 ModuleConfig 接口
2. 编写 useModuleRoutes hook，自动生成路由
3. 编写 useModuleMenus hook，自动生成菜单
4. 改造 App.tsx 和 AppLayout.tsx 使用上述 hooks
5. 把现有路由和菜单配置迁到 modules.ts
6. 编写模块扩展文档

**风险**：
- 需要改动核心路由，建议用功能分支并充分测试

### 2. 全量文案国际化

**建议步骤**：
1. 用脚本扫描代码，找出所有中文硬编码
2. 分批迁移到 uiText
3. 补充英文翻译
4. 用 grep 验证是否还有遗漏

**工具**：
```bash
# 找出所有中文字符
grep -r "[\u4e00-\u9fa5]" src/ --include="*.tsx" --include="*.ts"
```

### 3. 非 CRUD 模板化

**建议步骤**：
1. 分析 Chat、Agent、Document 页面的共同结构
2. 提取通用布局组件（SidebarLayout、EditorLayout 等）
3. 补充模板示例
4. 补充模板选择指南

**输出物**：
- `src/pages/Templates/SidebarLayout.tsx`
- `src/pages/Templates/EditorLayout.tsx`
- `src/pages/Templates/README.md`

### 4. 权限模型强化

**建议步骤**：
1. 补充权限模型设计文档（包括 RBAC、ABAC、行级权限的对比）
2. 扩展权限上下文，支持资源级权限
3. 补充权限初始化脚本
4. 补充权限清单（可用 Excel 或 JSON 格式）

**参考**：
```typescript
// 扩展权限上下文，支持资源级权限

  // 检查用户是否有对该资源的指定操作权限
```

## 总结

当前项目已经是"可靠的企业级可用基座"。它有清晰的分层、合理的数据流、统一的错误处理、基础的权限治理。但与"标准企业级中台"相比，还缺少以下几个方面的"规范化"：

1. **治理规范化**：模块、路由、菜单、权限的注册平台化
2. **国际化规范化**：全量文案集中管理
3. **模板规范化**：常见页面模式的标准化
4. **文档规范化**：缓存、错误、权限等核心策略的文档化

如果按上述路线推进，半年内可以达到"适合 10+ 人长期协作的企业级工程"。

不推荐一次性做完所有事，而是按"最小可用集合 → 持续迭代"的节奏。这样既能保证工程质量，也能及时收集实际使用反馈。
